# 📅 2주차 개발 체크리스트: LLM 연동 및 블로그 편집기 구현

## 프로젝트 기초 설정
- [ ] **전체 레이아웃 구성**: CSS Modules를 활용하여 메인 컨테이너, 헤더, 사이드바(또는 네비게이션)의 기본 레이아웃 구조를 잡습니다.
- [ ] **디렉토리 구조 설계**: Client/Server를 분리하고, 프론트엔드(`components/`, `hooks/`, `styles/`)와 백엔드(`routes/`, `utils/`) 도메인별 폴더 구조를 생성한 뒤 `gemini.md`를 배치합니다.

## 백엔드 및 GitHub API 연동
- [ ] **GitHub API 라우터 (Repository)**: Express에서 `.env`에 저장된 PAT(Personal Access Token)를 사용해 사용자의 Repository 목록을 안전하게 조회하는 API를 구현합니다.
- [ ] **브랜치/커밋 조회 API**: 프론트엔드에서 전달받은 레포지토리를 기반으로 브랜치 목록과 특정 브랜치의 커밋 로그를 가져오는 라우터를 구현합니다.
- [ ] **API 에러 핸들링**: GitHub API의 Rate Limit 초과 및 인증 에러(401, 403)를 처리하는 Express 미들웨어/예외 처리 로직을 추가합니다.

## 저장소 및 커밋 선택 UI 구현 (Frontend)
- [ ] **레포지토리 선택기**: 백엔드 API와 Native `fetch`로 통신하여 레포지토리 목록을 드롭다운이나 리스트 형태로 보여주는 UI를 구현합니다.
- [ ] **커밋 타임라인 뷰**: 선택된 레포지토리/브랜치의 커밋 로그를 날짜별로 나열하는 리스트 컴포넌트를 CSS Modules로 스타일링합니다.
- [ ] **커밋 다중 선택 (State)**: 체크박스를 통해 AI로 분석할 대상 커밋들을 다중 선택하고, 이를 React 상태(State)로 관리하는 로직을 추가합니다.

## 설계 및 검증 체계 (미션 필수 사항)
- [ ] **상태 흐름 설계**: AI와 협업하여 '선택된 커밋 -> 백엔드 -> LLM API -> 프론트 편집기'로 이어지는 데이터 흐름도와 Request/Response 구조를 작성하고 승인받습니다.
- [ ] **프롬프트 엔지니어링**: 커밋 로그를 블로그 초안(Markdown)으로 변환하기 위한 System Prompt를 최적화하고 응답 품질을 테스트합니다.
- [ ] **UX 엣지 케이스 도출**: 'AI 응답 지연(Timeout)', '커밋 내용이 부실할 때' 등의 예외 상황을 정의하고 에러 피드백 UI 및 로딩 처리 방안을 수립합니다.

## OpenAI 연동 및 데이터 보관
- [ ] **LLM 연동 라우터**: Express에서 커밋 데이터를 취합해 OpenAI `chat/completions` API로 전달하고, 마크다운 초안을 반환하는 API를 구현합니다.
- [ ] **로딩 애니메이션 (Spinner)**: LLM 응답 대기 시간(수십 초 소요 가능) 동안 사용자 경험을 해치지 않는 심미적인 로딩 컴포넌트를 구현합니다.
- [ ] **로컬 데이터 저장 (fs 모듈)**: Node.js 내장 `fs` 모듈을 사용하여 최종 생성된 블로그 데이터를 `posts.json` 파일에 읽고 쓰는(I/O) 로직을 백엔드에 구현합니다.

## 블로그 편집기 및 포스트 목록 뷰
- [ ] **마크다운 편집기(Editor)**: AI가 생성한 초안을 사용자가 직접 수정하고 보완할 수 있는 텍스트 에디터 인터페이스를 제공합니다.
- [ ] **포스트 카드 목록**: `posts.json`에서 불러온 저장된 포스트들을 카드 형태(브랜치 태그, 요약 미리보기, 날짜 포함)로 렌더링하는 뷰를 구현합니다.
- [ ] **수정 및 상태 변경**: 저장된 글을 다시 클릭하여 편집하거나, 최종 발행(Published) 여부를 토글할 수 있는 상호작용 인터페이스를 완성합니다.

## 개발 Workflow (매 항목 반복)
1. **설계**: 기능을 구현하기 전 컴포넌트 구조와 상태 관리를 먼저 정의한다.
2. **구현**: AI에게 구체적인 프롬프트를 전달하여 코드를 생성한다.
3. **리뷰**: AI가 작성한 코드를 한 줄씩 읽으며 의도를 파악하고 질문한다.
4. **검증**: 브라우저에서 디자인/기획 가이드와 일치하는지, 콘솔 에러는 없는지 확인한다.
5. **커밋**: `feat: #번호 기능명` 형식으로 커밋 로그를 남긴다.
